# 그룹 게임 컬렉션

여럿이 함께 즐기는 온라인 게임을 한곳에 모으는 웹 플랫폼입니다. 현재 첫 번째 게임으로 최대 30명이 그림을 공유하고 채팅으로 정답을 맞히는 `두들팡`을 제공합니다. Express가 정적 화면을 제공하고 Socket.IO가 방, 참가자, 그림, 채팅, 게임 상태를 동기화합니다.

향후 게임이 두 개 이상이 되면 루트 화면을 게임 선택 화면으로 전환하고, 두들팡과 새 게임을 각각 독립된 게임 진입점에서 실행할 예정입니다.

## 주요 기능

- 2~30명 공개/비공개 방 생성, 6자리 방 코드 입장
- 공개 방 검색, 입장 가능 필터, 최신순·인원순 정렬
- 방장 전용 시작, 설정 변경, 강퇴, 권한 위임, 방 종료
- 서버에서만 관리하는 제시어, 정답 판정, 점수, 타이머, 힌트
- 펜, 지우개, 색상, 굵기, 전체 지우기, 실행 취소·다시 실행
- 0~1 정규화 좌표와 24ms 배치 전송을 이용한 Canvas 동기화
- 서버 그림 동작 기록을 이용한 중도 입장·재접속 화면 복구
- 모바일 터치와 PC 포인터 지원, 모바일 하단 탭 UI
- 30초 재접속 유예, 자동 방장 위임, 빈 방 자동 삭제
- 채팅 100개 제한, 입력 속도 제한, 서버 권한·데이터 검증
- 라운드 결과, 전체 순위, 최다 정답·최다 출제 기록

효과음은 별도 저작권 파일 없이 Web Audio API의 짧은 톤으로 구현했습니다. 상단 스피커 버튼으로 음소거할 수 있습니다.

## 설치와 로컬 실행

Node.js 18 이상을 설치합니다. Node.js 공식 사이트의 LTS 버전을 권장합니다.

```bash
npm install
npm run dev
```

브라우저에서 `http://localhost:3000`에 접속합니다. 운영 방식으로 실행하려면 다음 명령을 사용합니다.

```bash
npm start
```

환경 변수:

| 이름 | 기본값 | 설명 |
|---|---:|---|
| `PORT` | `3000` | Express와 Socket.IO가 함께 사용할 포트 |

상태 확인 엔드포인트는 `GET /health`이며 `{ "ok": true, "rooms": 0 }` 형태로 응답합니다.

## 멀티플레이 테스트

두 브라우저 테스트:

1. 일반 창에서 닉네임을 입력하고 공개 방을 만듭니다.
2. 시크릿 창 또는 다른 브라우저에서 다른 닉네임으로 공개 방에 입장합니다.
3. 두 번째 사용자가 준비한 뒤, 방장이 게임 시작을 누릅니다.
4. 방장이 출제자와 직접 제시어를 설정합니다.
5. 출제자 화면에서만 그림이 조작되는지, 다른 화면에는 제시어가 노출되지 않는지 확인합니다.
6. 참가자 채팅에 정답을 입력해 점수와 라운드 결과를 확인합니다.
7. 플레이 중 한 브라우저를 새로고침해 30초 내 상태가 복구되는지 확인합니다.

자동화된 서버 통합 테스트:

```bash
npm test
```

테스트는 실제 Socket.IO 클라이언트 3개를 연결해 방 생성, 중복 닉네임, 비밀 제시어 전달 범위, 출제자 권한, 정답 정규화와 서버 점수를 확인합니다.

## 30명 테스트 방법

기본 검증은 브라우저 여러 개보다 Socket.IO 부하 도구를 별도 작성해 수행하는 것이 안정적입니다.

1. 테스트 전용 방을 최대 인원 30명으로 생성합니다.
2. `socket.io-client` 기반 스크립트에서 고유 `userId`와 닉네임을 가진 클라이언트 30개를 연결합니다.
3. 한 클라이언트만 초당 약 30~50개의 그림 배치를 보내고 나머지는 수신하게 합니다.
4. 모든 클라이언트에서 분당 채팅 수, 이벤트 지연, 메모리, CPU, 연결 해제를 기록합니다.
5. 30개 연결을 10분 이상 유지한 뒤 재접속과 라운드 전환을 반복합니다.

운영 환경에서는 `node --inspect` 또는 플랫폼 메트릭으로 메모리와 이벤트 루프 지연도 확인하세요. 현재 구현은 한 방 30명 규모를 목표로 하며, 전체 서버의 동시 방 수는 배포 인스턴스 사양과 그림 이벤트 빈도에 따라 달라집니다.

## Socket.IO 구조

주요 클라이언트 요청은 다음과 같습니다.

- `room:create`, `room:join`, `room:leave`: 방 수명 주기
- `room:ready`, `room:settings`, `room:transfer`, `room:kick`, `room:close`: 대기실과 방장 권한
- `game:start`, `game:restart`, `game:lobby`: 게임과 라운드 전환
- `chat:send`: 일반 채팅과 서버 정답 판정
- `canvas:draw`: 최대 30개 선분의 배치 전송
- `canvas:action`: 지우기, 취소, 다시 실행

서버가 보내는 핵심 이벤트는 `rooms:list`, `room:state`, `game:secret`, `game:hint`, `canvas:draw`, `canvas:sync`, `answer:correct`, `round:ended`입니다. `game:secret`은 방장과 현재 출제자의 소켓 ID로만 보냅니다. `room:state`나 공개 방 데이터에는 정답 문자열이 포함되지 않습니다.

Canvas 선분은 브라우저 크기와 무관하게 `fromX`, `fromY`, `toX`, `toY`를 0~1로 정규화합니다. 클라이언트는 포인터 이동을 약 24ms 간격으로 묶고, 서버는 현재 출제자인지와 좌표·색상·굵기를 다시 검사한 후 해당 Socket.IO room에만 중계합니다.

## 보안 주의사항

- 클라이언트가 보낸 방 코드, 사용자 ID, 닉네임, 설정, 그림 좌표를 모두 신뢰하지 않고 서버에서 검사합니다.
- 방장/출제자/참가자 권한과 현재 게임 상태가 맞지 않으면 요청을 거절합니다.
- 점수와 정답 판정은 서버만 변경합니다.
- 닉네임은 이모지와 특수문자를 허용하지만 길이와 제어문자를 제한하고, 화면에서는 `textContent`로 출력해 HTML을 실행하지 않습니다.
- 채팅은 사용자별 5초당 7개, 그림은 사용자별 초당 70개 요청으로 제한합니다.
- `userId`는 편의상 `localStorage`에 있는 임시 식별자입니다. 실제 서비스에서 계정 보안이 필요하면 로그인 세션, 서명된 토큰, CSRF/Origin 정책을 추가해야 합니다.
- 공개 인터넷에 배포할 때 HTTPS, 보안 헤더(예: Helmet), 프록시의 요청 제한과 모니터링을 권장합니다.

## 배포

GitHub Pages는 Node.js/Socket.IO 서버를 실행하지 못하므로 사용할 수 없습니다. 한 Node.js 프로세스가 프런트 파일과 실시간 서버를 함께 제공합니다.

### Render

1. 저장소를 GitHub에 올리고 Render에서 **New Web Service**를 만듭니다.
2. Build Command는 `npm ci`, Start Command는 `npm start`로 설정합니다.
3. Render가 제공하는 `PORT` 환경 변수를 서버가 자동 사용합니다.
4. 무료 인스턴스는 비활성 시 휴면과 첫 접속 지연이 있을 수 있습니다.

### Railway

1. Railway의 **Deploy from GitHub repo**로 저장소를 선택합니다.
2. 빌드와 시작 명령은 `package.json`에서 자동 감지됩니다.
3. Networking에서 공개 도메인을 생성합니다.
4. 여러 인스턴스로 확장하기 전에는 인스턴스 수를 1로 유지합니다.

### Fly.io

Node.js 앱용 `fly launch`로 설정을 만들고 내부 포트를 `3000`으로 지정한 뒤 `fly deploy`를 실행합니다. Fly가 주입하는 `PORT`를 쓰거나 설정 파일의 내부 포트와 맞춥니다. 다중 머신을 사용하려면 아래 Redis 구성이 먼저 필요합니다.

### 개인 VPS

Node.js LTS와 `npm ci --omit=dev`로 설치하고 PM2 또는 systemd로 `npm start`를 상시 실행합니다. Nginx/Caddy로 HTTPS를 종료하며 WebSocket 업그레이드 헤더를 전달해야 합니다.

## 알려진 제한 사항

- 방, 사용자, 점수와 그림 기록은 프로세스 메모리에 있으므로 서버 재시작 시 사라집니다.
- 단일 Node.js 인스턴스를 기준으로 합니다. 여러 인스턴스에 임의 분산하면 서로 다른 인스턴스의 참가자가 같은 방을 공유하지 못합니다.
- 그림 동작 기록은 현재 라운드만 보관하고 액션 수를 제한합니다. 매우 긴 라운드에서 제한을 넘긴 초기 획은 재접속 화면에서 제외될 수 있습니다.
- 30초 재접속 유예 중 프로세스가 재시작되면 복구할 수 없습니다.
- 실제 음원 파일, 사용자 계정, 신고/차단, 관전자 전용 모드, 영구 전적은 포함하지 않았습니다.
- 본격적인 인터넷 부하 테스트는 배포 환경의 CPU, 메모리, 네트워크 조건에서 별도로 수행해야 합니다.

## Redis와 데이터베이스 확장

`lib/memory-room-store.js`는 방 저장소 경계를 제공합니다. 영구 확장 시 다음 순서를 권장합니다.

1. 방 설정, 사용자, 게임 결과 같은 영구 데이터는 PostgreSQL 등에 저장합니다.
2. 현재 라운드, 접속 상태, 정답자 집합, 재접속 TTL은 Redis에 저장합니다.
3. `MemoryRoomStore`와 동일한 책임을 가진 `RedisRoomStore`를 만들고 원자적 갱신은 Redis 트랜잭션 또는 Lua로 처리합니다.
4. Socket.IO Redis adapter를 붙여 여러 Node.js 인스턴스 사이에서 room 이벤트를 공유합니다.
5. 로드 밸런서에서 WebSocket을 허용하고, 가능하면 sticky session을 설정합니다.
6. 정답은 로그나 일반 API 응답에 남기지 말고 최소 수명으로 보관합니다.

Redis로 전환하더라도 점수 계산과 권한 검사는 계속 서버의 단일 규칙 계층에서 수행해야 합니다.
